# ADR-058 — Генерация изображений: `generated_images`, `GET /v1/images/{id}`, отдельный идемпотентный дебет

- **Статус:** Accepted
- **Дата:** 2026-07-09
- **Связано:** [ADR-011](ADR-011-server-side-tools.md) (server-side tool в tool-loop), [ADR-033](ADR-033-llm-provider-abstraction.md) (провайдер-абстракция LLM — генератор изображений **вне** её, независим от `LLM_PROVIDER`, §3), [ADR-055](ADR-055-dialog-mode.md) (режимы диалога), [ADR-059](ADR-059-openai-default-provider.md) (OpenAI-провайдер, `images.generate` контракт), [ADR-006](ADR-006-credit-billing-and-subscription-grant.md) (1 кредит = 1 сообщение для текста; image-дебет — отдельный, §4), [ADR-002](ADR-002-access-policy-state-machine.md) (trial/policy — image требует кредитов, §4), [ADR-044](ADR-044-multi-provider-byok.md) (byok-генерация текста бесплатна — image исключение, §4), [ADR-005](ADR-005-idempotency-ledger.md) (idempotency ledger), [ADR-004](ADR-004-blocked-http-200.md) (blocked+HTTP 200, blockReason), [ADR-010](ADR-010-backend-hosted-preview.md) (BYTEA-хранение, отдача байтов с security-заголовками), [ADR-025](ADR-025-parallel-tool-calls-and-max-tokens-truncation.md) (реплей continuation → защита от дубля), [ADR-056](ADR-056-temporary-chat.md) (session_id=None у временного чата)

## Контекст

Требуется генерация изображений: модель по запросу пользователя создаёт картинку, которая **хранится вне чата** и отдаётся отдельным endpoint для фетча (в `chat_steps.payload` — только ссылка, не байты). Anthropic изображения **не генерирует вообще**; генерацию даёт OpenAI (`images.generate`, модель `gpt-image-1`, [ADR-059](ADR-059-openai-default-provider.md)). Картинки — дорогой ресурс, поэтому нужен отдельный биллинг сверх обычного кредита за ход.

## Решение

### 1. Таблица `generated_images` (BYTEA), в чате — только id

Байты изображения хранятся в новой таблице `generated_images` (BYTEA, по образцу `site_files` [ADR-010](ADR-010-backend-hosted-preview.md)/`workspace_files` [ADR-036](ADR-036-workspaces-implementation.md)). В `chat_steps.payload` кладётся **только `id`** изображения, **никогда байты** — история остаётся лёгкой, тяжёлый контент не реплеится в tool-loop (симметрия с плейсхолдерами вложений [ADR-020](ADR-020-inline-base64-attachments-mvp.md)).

Колонки: `id` (UUID PK), `user_id` (NOT NULL, FK users CASCADE — **единственная опора авторизации** в GET), `session_id` (NULL, FK chat_sessions SET NULL — `NULL` для временного чата [ADR-056](ADR-056-temporary-chat.md)), `message_step_id` (NULL, корреляция с ходом), `tool_call_id` (NULL, ключ идемпотентности дебета), `content` (LargeBinary NOT NULL, BYTEA), `content_type`, `size` (`CHECK (size >= 0)`), `prompt`, `created_at`, **`expires_at` (TIMESTAMPTZ NULL — §6, TTL временного чата; `NULL` = не истекает, для обычного чата)**. Индексы: `(user_id, created_at)`; **частичный UNIQUE** на `tool_call_id WHERE tool_call_id IS NOT NULL` — защита от дубля строки при реплее continuation ([ADR-025](ADR-025-parallel-tool-calls-and-max-tokens-truncation.md)); **частичный индекс `(expires_at) WHERE expires_at IS NOT NULL`** — быстрый отбор истёкших строк для sweep (§6). **`expires_at` вводится миграцией `0016` СРАЗУ** (не добавляется отдельной поздней миграцией) — фича временного чата ([ADR-056](ADR-056-temporary-chat.md)) уже в проде к моменту спринта 5.

### 2. `GET /v1/images/{imageId}` под JWT; чужое → 404

Отдельный endpoint фетча `GET /v1/images/{imageId}` под JWT. Изображение другого пользователя → **404, не 403** — не раскрываем существование чужого ресурса (как `GET /v1/chats/{id}` и preview [ADR-010](ADR-010-backend-hosted-preview.md)). Авторизация — по `generated_images.user_id == sub`. Отдаётся `Response(content=..., media_type=...)` с security-заголовками по образцу `preview.py` (`X-Content-Type-Options: nosniff`, `Cache-Control: private, no-store`); CSP-sandbox для картинок не нужен (не активный контент).

**Истёкшее изображение (TTL, §6) → тоже `404`, не раскрывая существования.** Запрос фетча содержит **условие свежести**: `WHERE id=:id AND user_id=:sub AND (expires_at IS NULL OR expires_at > now())`. Истёкшее изображение (`expires_at <= now()`) неотличимо для клиента от чужого/несуществующего — **`404`**. **Ключевой инвариант: логическая недоступность проверяется по `expires_at` в запросе, НЕ по факту физического удаления байт** — даже если sweep (§6) ещё не удалил строку, истёкшее изображение уже нечитаемо. Так приватность временного чата не зависит от наличия планировщика.

### 3. Server-side tool `image.generate`; OpenAI отдельным клиентом

`image.generate` — server-side tool в tool-loop ([ADR-011](ADR-011-server-side-tools.md)): исполняется backend, наружу как `tool_call` не отдаётся. Провайдер — OpenAI `gpt-image-1` через **отдельный клиент** (`ImageGenerator`/`OpenAIImageGenerator`), а **не** через `LLMClient` и **не** через встроенный Responses-tool `ImageGeneration`. Обоснование отдельного клиента: нужен контроль над **байтами** (сохранить в БД) и над **биллингом** (отдельный дебет), которого встроенный tool не даёт. Для `gpt-image-1` параметр `response_format` API отвергает — результат всегда base64 → `b64decode` (`output_format` управляет форматом; [ADR-059](ADR-059-openai-default-provider.md)).

**Критерий активности инструмента — конфигурация генератора, а НЕ `LLM_PROVIDER` (ревизия по итогам спринта 5, исправляет прежнюю формулировку «на Anthropic не активен — нет провайдера генерации»).** `OpenAIImageGenerator` владеет **собственным** `AsyncOpenAI(api_key=openai_api_key)` с image-специфичными таймаутами/ретраями и **не зависит** от `LLM_PROVIDER` / провайдер-абстракции [ADR-033](ADR-033-llm-provider-abstraction.md) (`image_client.py`; он не `LLMClient`). Провайдер генерации изображений **изначально отдельный** от провайдера чата — Anthropic картинки не генерирует **вообще**, поэтому генератор всегда OpenAI. Значит **Anthropic-чат-инстанс с заданным `OPENAI_API_KEY` способен генерировать изображения** — привязывать фичу к оси `LLM_PROVIDER` неверно. **Решение: `image.generate` предлагается модели тогда и только тогда, когда задан ключ генерации (`OPENAI_API_KEY` непуст)** — новый, **четвёртый** offered-set гейт в `tools.py` (рядом с `include_server_side` / `include_client_side` / `dialog_mode`, композируется по И). **Отклонено:** гейт по `LLM_PROVIDER` (лишил бы Anthropic-инстанс с ключом рабочей функции без технической причины); «не гейтить вовсе» (модели показывался бы заведомо-падающий при отсутствии ключа инструмент → лишние токены/tool-раунды, а ход всё равно тарифицируется). Если оператор захочет отключать картинки при заданном `OPENAI_API_KEY` — отдельный флаг `IMAGE_GENERATION_ENABLED` (будущее уточнение, не требуется сейчас). *Гейт по конфигу статичен; degrade (ниже) — safety-net для рантайм-сбоя, когда ключ **задан**, но вызов упал.*

**Обработка ошибок генератора — degrade по ВСЕМ классам, инфра-сбой логируется (ревизия по итогам спринта 5; прежде §3 описывал только content-policy).** Ни один класс ошибок генератора не роняет ход `502`/`422` и не создаёт `blocked` — все деградируют в `tool_result`-ошибку (ход выживает, [ADR-026 §6](ADR-026-global-server-side-tools-and-time-now.md)-паттерн). **Биллинг при любом сбое генерации:** `IMAGE_CREDITS_COST` **НЕ** списывается (генерация падает ДО `consume`, §5 — «нет байт → нет image-дебета»); **кредит за ход (1) списывается как обычно**, т.к. текстовый ход состоялся (пользователь не платит отдельным дебетом за несозданную картинку, но платит за ход). Классы (по `image_client.py`):

**⚠️ Дискриминатор класса `BadRequestError` — поле `code`, а НЕ `type` (подтверждено живой проверкой gpt-image-1 против боевого API, спринт 5).** `gpt-image-1` возвращает **три** разных отказа **все** как `openai.BadRequestError`, различающихся **только полем `exc.code`**: `moderation_blocked` (отказ модерации — user-facing), `invalid_value` (неверное значение параметра, напр. `size`), `unknown_parameter` (неизвестный параметр — наша ошибка вызова). **Поле `type` НЕ годится как дискриминатор** — у модерации и у неверного `size` оно **совпадает**; выбор `type` молча смешал бы «модель нарушила политику» с «мы передали кривой параметр». Поэтому `ImageContentPolicyError` поднимается **только при `exc.code == "moderation_blocked"`**; прочие `BadRequestError`-коды (`invalid_value`/`unknown_parameter`) — это **сбой/misconfig**, а не content-policy, и идут в `ImageGenerationError` (класс «сбой генерации», логируется WARNING). Это боевое знание — следующий разработчик, выбрав `type`, сломает классификацию молча.

| Класс ошибки | Источник (по `image_client.py`) | Поведение | Логирование |
|---|---|---|---|
| **content-policy** | `ImageContentPolicyError` — **только** `openai.BadRequestError` с `exc.code == "moderation_blocked"` | `tool_result`-ошибка с кодом `content_policy`; модель **корректирует промт в том же ходе** и повторяет (graceful degrade, как `time.now`/`invalid_timezone`) | INFO/DEBUG — нормальный user-facing исход, не инцидент |
| **сбой генерации** | `ImageGenerationError` — `BadRequestError` с **прочим** `code` (`invalid_value`/`unknown_parameter` — кривой параметр/наша ошибка вызова), а также таймаут/сеть/5xx `OpenAIError`, oversize, no-data, invalid-base64 | `tool_result`-ошибка с кодом `image_generation_failed`; ход выживает (транзиентный сбой image-провайдера **не** должен `502`-ить чат) | **WARNING + метрика** — инфра-сигнал **НЕ теряется** (закрывает контраргумент «degrade скрывает сбой в мониторинге»); **без промпта и ключа** ([TD-035](../100-known-tech-debt.md)/[05-security.md](../05-security.md#логирование-безопасное)) |
| **нет/невалиден ключ** | нет ключа → гейт выше **не предлагает** инструмент (класс не возникает); ключ задан, но невалиден/отозван → `OpenAIError` → `ImageGenerationError` | как «сбой генерации» (degrade) | **WARNING** «misconfig/invalid image key» |

**Обоснование degrade (а не 502):** текстовый ответ хода ценен сам по себе, изображение — дополнение; на инстансе без ключа `image.generate` не предлагается (гейт §3), а при заданном-но-сбойном ключе транзиентный сбой не должен ронять весь чат. Контраргумент «потеря сигнала для мониторинга» снят обязательным структурным логированием инфра-класса (симметрично [TD-014](../100-known-tech-debt.md) для Anthropic upstream-ошибок: наружу degrade, внутрь — лог).

### 4. Отдельный идемпотентный дебет `IMAGE_CREDITS_COST` — списывается ВСЕГДА, независимо от `mode` (решение пользователя, ревизия по итогам спринта 5)

Изображение списывает `IMAGE_CREDITS_COST` кредитов **дополнительно** к биллингу хода. Дебет — через существующий `wallet.consume(idempotency_key=str(tool_call_id))` ([ADR-005](ADR-005-idempotency-ledger.md)): ключ = `tool_call_id`, поэтому повтор (реплей continuation [ADR-025](ADR-025-parallel-tool-calls-and-max-tokens-truncation.md)) не спишет дважды. wallet-код не трогаем.

**`IMAGE_CREDITS_COST` списывается ВСЕГДА при успешной генерации изображения — независимо от `mode` (`credits` ИЛИ `byok`).** Прежняя формулировка «дебет только при `credits`; для `byok`/trial картинка бесплатна» **пересмотрена** (решение пользователя). **Почему:** изображение **всегда** генерируется на **серверном** `OPENAI_API_KEY` (§3, отдельный генератор вне провайдер-абстракции) — расход несём **мы**, в отличие от текста, где BYOK-пользователь платит **своим** ключом. Поэтому:
- **`mode=credits`, активная подписка:** ход = 1 кредит (текст, [ADR-006](ADR-006-credit-billing-and-subscription-grant.md)) **+** `N × IMAGE_CREDITS_COST` (каждое изображение). Итог: `1 + N×COST`.
- **`mode=byok`:** текст — бесплатно (ключ пользователя, `_billing_plan` не списывает), **но** `N × IMAGE_CREDITS_COST` за изображения **списывается** (наш ключ). Итог: `0 (текст) + N×COST`. BYOK **не освобождает** от оплаты изображений.

**Не противоречит «byok-генерация бесплатна» ([ADR-044 §6](ADR-044-multi-provider-byok.md)/[ADR-006](ADR-006-credit-billing-and-subscription-grant.md)):** та формулировка относится к **генерации текста через ключ пользователя** (`_billing_plan`, turn-billing). Изображение — **не** «byok-генерация»: оно идёт через **наш** серверный ключ, поэтому это отдельная, **mode-независимая** операция с собственным дебетом. `_billing_plan` (turn-billing) не меняется; image-дебет живёт **вне** него (в tool-handler).

**Trial (решение — вариант (а)): trial даёт бесплатный ТЕКСТ, но изображение требует кредитов.** Пробный пользователь (`subscription=none`, `trial_used=false`) получает бесплатный текстовый ход (flip `users.trial_used`, [ADR-002](ADR-002-access-policy-state-machine.md)), но **изображение требует ненулевого баланса кредитов**: `wallet.consume(IMAGE_CREDITS_COST)` при нулевом балансе бросит `InsufficientCreditsError` → ход `blocked` + `image_credits_empty` (§5). Выбор (а) обоснован: (1) последователен с принятым решением «за изображение платим кредитами, т.к. генерируем на нашем ключе»; (2) **не требует новой логики** — путь `InsufficientCreditsError`→`image_credits_empty` уже реализован; (3) trial-ход с изображением при нуле баланса откатывается **до** финального биллинга (§5, early-return из tool-loop), поэтому **trial НЕ потребляется** (flip не достигается) — пользователь может повторить текстом-без-картинки и сохранить свой trial. Вариант (б) («trial включает N бесплатных изображений») отклонён: вводит отдельный бесплатный лимит на нашем ключе (абуз-вектор), новую логику подсчёта, и расходится с принципом «изображение = наш расход = кредиты».

### 5. Порядок: сначала `consume`, потом `INSERT`; при нехватке — rollback, байты не сохраняются

Порядок операций критичен:
1. получить байты от `ImageGenerator`;
2. **`consume`** `IMAGE_CREDITS_COST` (может бросить при нехватке);
3. при успехе — **`INSERT generated_images`**; в `chat_steps.payload` — только id.

Сначала `consume` (списание может отказать), затем `INSERT` (иначе байты сохранились бы без оплаты). При **нехватке кредитов** — байты **не сохраняются**, `rollback`, ход возвращает `blocked` + **новый `blockReason: image_credits_empty`** (HTTP 200, [ADR-004](ADR-004-blocked-http-200.md)). Новый blockReason (а не переиспользование `credits_empty`) — чтобы UI различал «кончились кредиты на генерацию картинки» от «кончились кредиты на сообщение» и показал профильный CTA. Действует **и для `byok`, и для trial** (§4): нехватка баланса под `IMAGE_CREDITS_COST` блокирует картинку в любом режиме.

**Неизбежное следствие порядка `generate → consume → INSERT`: при `image_credits_empty` байты УЖЕ сгенерированы (и оплачены НАМИ на `OPENAI_API_KEY`), но НЕ сохранены и пользователю НЕ отданы.** `consume` идёт **после** генерации (иначе списали бы за неудачную генерацию — content-policy/сбой могли бы вернуть ошибку, §3), поэтому на момент проверки баланса генерация уже совершена и наш upstream-расход понесён. При недостатке кредитов строка `generated_images` не вставляется, транзакция откатывается, `images[]` пуст — пользователь ничего не получает, а сгенерированные байты отбрасываются. Это **осознанный компромисс** порядка: списывать до генерации нельзя (иначе брали бы плату за неуспешную генерацию). Ущерб ограничен: rate-limit + блокировка не дают спамить (каждая попытка при нуле баланса → `blocked`, картинка не отдаётся); при активной подписке с балансом кейс не возникает. Абуза «бесплатных генераций на нашем ключе» это **не** создаёт — пользователь не получает изображение, лишь провоцирует единичный отбрасываемый вызов под rate-limit.

### 6. Изображения временного чата: TTL + удаление без планировщика (решение пользователя)

**Контекст.** `image.generate` во временном чате ([ADR-056](ADR-056-temporary-chat.md)) пишет байты в `generated_images`, и картинка **переживает** диалог — это противоречит «временности». **Решение пользователя: генерацию во временном чате РАЗРЕШИТЬ, но изображения временного чата автоматически удалять по TTL.** Ограничение проекта: **ни планировщика, ни cron нет** ([TD-010](../100-known-tech-debt.md)/[TD-013](../100-known-tech-debt.md)) — «автоудаление» строим только на Postgres + Redis.

**Схема.** Колонка `generated_images.expires_at` (§1):
- **обычный чат** → `expires_at = NULL` (не истекает);
- **временный чат** → `expires_at = created_at + TEMPORARY_IMAGE_TTL_SECONDS` (дефолт **86400** = 24 ч; конфигурируемо). Дефолт 24 ч принят как разумный компромисс: достаточно, чтобы пользователь успел сохранить/просмотреть картинку в рамках сессии, и достаточно коротко для «временности».

**Логическая недоступность (гарантирована, не зависит от GC).** `GET /v1/images/{id}` истёкшего изображения → `404` по условию `expires_at > now()` в запросе (§2). Это **безусловная** гарантия приватности: истёкшая картинка нечитаема сразу по времени, **независимо** от того, удалены ли байты физически.

**Физическое удаление байт — opportunistic sweep (выбранный механизм).** Без планировщика периодический GC сам не запустится. Рассмотрены два варианта:

| Вариант | Как | Плюсы | Минусы |
|---|---|---|---|
| **A. Opportunistic sweep (выбран)** | На некоторых request-путях (генерация изображения / фетч `GET /v1/images/{imageId}` / список `GET /v1/images`, §8 / удаление `DELETE /v1/images/{imageId}`, §9) backend удаляет **малую партию** истёкших строк: `DELETE FROM generated_images WHERE expires_at IS NOT NULL AND expires_at <= now() ... LIMIT IMAGE_SWEEP_BATCH_SIZE`, под **Redis-локом** (одновременно метёт максимум один воркер) и **троттлингом** (Redis-ключ с TTL `IMAGE_SWEEP_MIN_INTERVAL_SECONDS` — не чаще раза в интервал). | Самозапускается от трафика (планировщик не нужен); ограниченное влияние на latency (один воркер, малый батч, rate-limit); без новой инфраструктуры; согласуется с «строим на Postgres+Redis». | Не гарантирован: на инстансе без image-трафика истёкшие (уже нечитаемые) байты не выметаются → лежат до появления трафика. |
| **B. One-shot compose-задача** | Отдельный `sweeper`-сервис в `docker-compose` рядом с `migrate`, запускается вручную/на деплое: `DELETE ... WHERE expires_at <= now()`. | Явный, предсказуемый разовый прогон; не бьёт по request-latency. | **Не периодичен** — между деплоями/ручными запусками ничего не метёт; без cron не даёт регулярности; требует операторского действия. |

**Выбор — A (opportunistic sweep)**, дополнительно **B как опциональный операторский инструмент** для форс-очистки (документируется, не обязателен). Обоснование: A **самозапускается** от реального трафика (единственный способ регулярной уборки без планировщика), а его влияние на latency ограничено Redis-локом + троттлингом + малым батчем; B в одиночку не даёт регулярности (одноразов, нужен внешний триггер). Логическая гарантия (§2, 404 по `expires_at`) от выбора механизма **не зависит** — она безусловна.

**Остаточный долг.** Ни A, ни B без планировщика не дают **гарантированного** периодического удаления: на инстансе с низким image-трафиком истёкшие байты (уже логически недоступные) могут оставаться до появления трафика или ручного прогона B. Это реальный остаточный долг — **[TD-036](../100-known-tech-debt.md)** (best-effort физический sweep без планировщика; логическая приватность обеспечена безусловно). Связано с [TD-010](../100-known-tech-debt.md)/[TD-013](../100-known-tech-debt.md) (отсутствие планировщика) и [TD-031](../100-known-tech-debt.md) (BYTEA vs object storage).

**Config (§6):** `TEMPORARY_IMAGE_TTL_SECONDS` (дефолт `86400`), `IMAGE_SWEEP_BATCH_SIZE` (размер партии удаления), `IMAGE_SWEEP_MIN_INTERVAL_SECONDS` (троттлинг sweep). Malformed → безопасный дефолт (стиль `resolved_*`).

### 7. Форма ответа `images[]` — только id и метаданные, без промпта (приватность)

Поле `images[]` в `ChatResponse` содержит по каждому изображению **строго** `imageId`, `contentType`, `size` — **без `prompt`** и без байтов. Промт **не** возвращается клиенту: defense-in-depth ([TD-035](../100-known-tech-debt.md)) — свободный текст (промт) не эхоится в ответ, который может персиститься/логироваться. Промт генерации хранится server-side в `generated_images.prompt` (для отладки/регенерации), но **не** попадает ни в ответ, ни в логи. Байты не в ответе — фетч `GET /v1/images/{imageId}` (§2). Контракт: `images: [{ imageId, contentType, size }]`.

### 8. `GET /v1/images` — список изображений пользователя (курсорная пагинация) (ревизия 2026-07-10, решение пользователя)

Endpoint списка **всех** изображений владельца под JWT. Дополняет фетч `GET /v1/images/{imageId}` (§2): даёт клиенту перечень его сгенерированных картинок (галерея) без обхода истории каждого чата. Роутер — в существующем `src/app/api_gateway/routers/images.py` (тот же тег `Images`).

**Изоляция и свежесть — те же инварианты, что в §2 (не ослаблять).** Выборка строго `WHERE user_id = :sub AND (expires_at IS NULL OR expires_at > now())`. Владелец-only (`user_id == sub` — единственная опора авторизации, §1); истёкшие по TTL (§6) изображения в списке **не видны** — та же логическая недоступность, что в item-эндпоинте (по `expires_at` **в запросе**, не по факту физического sweep). Чужое/истёкшее/несуществующее не даёт отдельного кода — оно просто отсутствует в выдаче (пустой/усечённый список, `200`).

**Форма ответа — конверт `{ items, nextCursor }` (как `GET /v1/chats` и workspaces).** Envelope-имя — **`items`**, НЕ `images`: консистентность со всеми list-эндпоинтами проекта (chats, workspaces отдают `{ items, nextCursor }`), клиент переиспользует общий парсер списка. *(Отклонено `images` как имя массива: локальная описательность не оправдывает расхождение конверта списков; поле `images[]` в `ChatResponse` — иной контекст, inline-поле ответа генерации, а не пагинируемый конверт, конфликта нет.)* Каждый элемент `ImageListItem` — **строго** `{ imageId, contentType, size, createdAt, sessionId }`:
- `imageId` (= `generated_images.id`), `contentType` (= `content_type`), `size` (байты, `= size`), `createdAt` (ISO8601, `= created_at`);
- `sessionId` (= `generated_images.session_id`) — **nullable**: `null` для изображения временного чата (session `SET NULL`, §1/[ADR-056](ADR-056-temporary-chat.md)).
- **Без `prompt`** (приватность/[TD-035](../100-known-tech-debt.md) — сохраняем §7-инвариант: промт хранится server-side, наружу не отдаётся ни в `images[]` генерации, ни в списке) и **без байтов `content`** (байты — только фетч §2).

**Порядок и курсор.** Сортировка `created_at DESC, id DESC` (новые первыми; `id` — стабильный tie-break при равном `created_at`). Пагинация — **opaque keyset-курсор**, механизм зеркалит `ChatCursor` (`src/app/chats/cursor.py`): base64url кортежа последней строки страницы (`created_at.isoformat()` + разделитель + `id`), декод-ошибка → `InvalidCursorError` → **`422`** (как в chats). Keyset-предикат «строго после курсора»: `created_at < :c_created OR (created_at == :c_created AND id < :c_id)`. Fetch `limit + 1` для вычисления `nextCursor` без второго count-запроса (паттерн `ChatsRepository.list_chats`). `nextCursor = null`, когда следующей страницы нет.

**Параметры.** `cursor` (opaque, опц.), `limit` (int, `1..100`, дефолт **30** — как `GET /v1/chats` `_LIST_LIMIT_DEFAULT`). `limit` вне диапазона или битый `cursor` → **`422`**. Endpoint под тем же rate-limit-контуром, что list-чатов (`enforce_other_limits` → `429`).

**Opportunistic sweep (§6) — list-хендлер тоже триггер.** `GET /v1/images` вызывает `maybe_sweep_expired_images` (`src/app/chat/image_sweep.py`) так же, как item-эндпоинт §2: best-effort физический GC истёкших строк под Redis-локом + троттлингом, own tx, **fail-open** — sweep **никогда** не валит запрос списка. На корректность выдачи он не влияет: невидимость истёкших картинок гарантирует **предикат свежести `expires_at` в самом запросе** (выше), а не факт физического удаления ([TD-036](../100-known-tech-debt.md)).

**Индекс достаточен — новой миграции НЕТ.** Существующий `ix_generated_images_user_created` на `(user_id, created_at DESC)` (§1) покрывает ведущие колонки выборки (`user_id = :sub`) и сортировки (`created_at DESC`). `id`-tie-break внутри равных `created_at` (коллизии timestamptz редки) досортировывается поверх индексного скана — этого достаточно для эффективной keyset-пагинации; расширять индекс до `(user_id, created_at DESC, id DESC)` **не требуется** (не долг, не блокер — микрооптимизация на будущее). Таблица и оба индекса уже существуют → **отдельная миграция не нужна**.

**Коды.** `200` (в т.ч. пустой `items: []`); `401` (нет/невалидный JWT); `422` (`limit` вне диапазона / битый `cursor`); `429`; `5xx`. `404` для списка не применяется (пустая выдача — не ошибка).

### 9. `DELETE /v1/images/{imageId}` — удаление изображения из галереи (амендмент 2026-07-14, решение пользователя)

Endpoint удаления **своего** сгенерированного изображения из списка (§8). Тот же роутер `src/app/api_gateway/routers/images.py`, тег `Images`. Мотивация: пользователю нужно убрать картинку из галереи (`GET /v1/images`); физически это **hard-delete** строки `generated_images` — освобождается тяжёлый BYTEA `content` (соотносится с [TD-031](../100-known-tech-debt.md) BYTEA-хранение). Пользовательский `DELETE` освобождает BYTEA **только у НЕистёкших** картинок (обычный чат, `expires_at IS NULL`): предикат свежести `expires_at > now()` в запросе (ниже) **не матчит истёкшие** строки, поэтому истёкшие (картинки временного чата — предмет [TD-036](../100-known-tech-debt.md)) `DELETE` **НЕ** удаляет; их физически сносит opportunistic sweep (§6), триггерируемый в т.ч. этим DELETE-хендлером. Таким образом это лишь **частичная**, инициируемая пользователем компенсация отсутствия гарантированного GC [TD-036](../100-known-tech-debt.md), а не его замена.

**Изоляция и свежесть — те же инварианты, что §2/§8 (не ослаблять).** Удаление **одним** стейтментом с owner + freshness **В ЗАПРОСЕ**: `DELETE FROM generated_images WHERE id = :id AND user_id = :sub AND (expires_at IS NULL OR expires_at > now())`. `user_id == sub` — единственная опора авторизации (§1). Чужое / истёкшее по TTL (§6) / несуществующее → удалено **0 строк** → **`404`**, **не раскрывая существования** (консистентно с `GET /v1/images/{imageId}` §2, `404`-for-foreign; **не `403`** — owner-условие в запросе, а не сверка body-`userId`). Истёкшее логически недоступно (§6) — его удаление тоже `404` (владелец его «не видит» → «не может удалить»), а физические байты выметет sweep (ниже). **Ключевой инвариант-parity: DELETE не отдаёт `204` на то, на что GET отдаёт `404`** — свежесть проверяется тем же предикатом `expires_at`, иначе delete раскрыл бы существование логически-недоступной строки.

**Успех → `204 No Content`** (удалено 1 строка). Тело пустое: ресурс удалён, отдавать нечего — строже выражает «контента нет», чем envelope `{deleted:true}` (у images нет прочих мутаций с телом-конвертом, ре­юз общего парсера не требуется; `DELETE /v1/chats/{id}` отдаёт `{deleted:true}` из-за исторического контракта chats, здесь его не тиражируем). `_SECURITY_HEADERS` (`X-Content-Type-Options: nosniff` + `Cache-Control: private, no-store`) — на ответах **и `204`, и `404`** (как в §2).

**Идемпотентность/повтор → `404` (НЕ `204`).** Повторный `DELETE` уже удалённого (либо чужого, либо истёкшего) → строки нет → удалено 0 → **`404`**. Осознанно **не** идемпотентно-`204`: `204` на отсутствующую строку косвенно подтвердил бы, что она когда-то существовала (или существует у другого) — расходится с §2-инвариантом «не раскрываем существование». `404`-on-missing консистентен с `DELETE /v1/chats/{id}` (повторное удаление → `404`).

**Побочные ссылки НЕ каскадируют в историю (ожидаемое поведение).** `chat_steps.payload` хранит **только `id`** изображения (§1), не байты. После удаления этот `id` становится «висячим» (dangling): последующий `GET /v1/images/{imageId}` по нему → `404` — **это ожидаемо и допустимо**. Историю чата (`chat_steps`/`tool_calls`) удаление изображения **НЕ трогает** (нет обратного FK image→step; ход остаётся в истории со ссылкой на удалённую картинку). Пользователь удаляет артефакт из галереи, **не** редактируя переписку; чистка истории — отдельные операции (`DELETE /v1/chats/{id}` / edit [ADR-040](ADR-040-edit-message-and-regenerate.md)).

**Биллинг — no-refund-on-delete.** Удаление изображения **НЕ возвращает** `IMAGE_CREDITS_COST` (симметрично no-refund-on-edit [ADR-040](ADR-040-edit-message-and-regenerate.md)): дебет за генерацию (§4) финализирован при создании, удаление артефакта его не отменяет.

**Opportunistic sweep (§6) — DELETE-хендлер тоже триггер.** Перед удалением вызывается `maybe_sweep_expired_images` (`src/app/chat/image_sweep.py`) — как §2/§8: best-effort физический GC истёкших строк под Redis-локом + троттлингом, own tx, **fail-open** (никогда не валит запрос). На корректность delete не влияет: целевая строка сносится по собственному предикату независимо от sweep.

**Rate-limit** — тот же контур, что GET/list (`enforce_other_limits` → `429`).

**Индекс достаточен — новой миграции НЕТ.** DELETE идёт по PK (`id`) + `user_id` — покрыто существующими индексами (§1). Таблица `generated_images` уже существует → **отдельная миграция не нужна**. Аудит-событие не вводится (симметрично `DELETE /v1/chats/{id}`).

**Коды.** `204` (удалено); `401` (нет/невалидный JWT); `404` (чужое / истёкшее / несуществующее / повторное); `429`; `5xx`. `403` не применяется.

## Альтернативы

- **Хранить картинку прямо в `chat_steps.payload` (base64).** Отклонено: раздувает историю, тяжёлый контент реплеится в каждом tool-витке; отдельная таблица + id в payload — лёгкая история (симметрия [ADR-020](ADR-020-inline-base64-attachments-mvp.md)).
- **Встроенный Responses-tool `ImageGeneration`.** Отклонено: не даёт контроля над байтами (сохранить в БД под наш id) и над биллингом (отдельный дебет); отдельный `images.generate`-клиент даёт и то, и другое.
- **`403` на чужое изображение.** Отклонено: раскрывает существование чужого ресурса; `404` консистентно с `chats`/`preview` ([ADR-010](ADR-010-backend-hosted-preview.md)).
- **Object storage (S3) вместо BYTEA.** Отклонено на MVP: новая инфраструктура; BYTEA транзакционен с ходом и согласуется с `site_files`/`workspace_files` ([TD-009](../100-known-tech-debt.md)/[TD-027](../100-known-tech-debt.md)). Вынос — [TD-031](../100-known-tech-debt.md).
- **Переиспользовать `blockReason=credits_empty` при нехватке на картинку.** Отклонено: UI не отличит причину; отдельный `image_credits_empty` даёт профильный CTA.
- **`INSERT` до `consume`.** Отклонено: при отказе списания байты остались бы неоплаченными; порядок consume→INSERT + rollback гарантирует «нет оплаты — нет строки».
- **Один кредит на ход, без доп-дебета за картинку.** Отклонено: генерация изображения существенно дороже текстового хода; отдельный `IMAGE_CREDITS_COST` защищает себестоимость (в отличие от выбора модели [TD-025](../100-known-tech-debt.md), здесь разница в стоимости качественная).
- **Запретить генерацию изображений во временном чате.** Отклонено (решение пользователя): режим полезен и во временном чате; вместо запрета — TTL + автоудаление (§6).
- **Удалять байты картинки временного чата сразу после хода (синхронно).** Отклонено: пользователь не успеет получить картинку (`GET /v1/images` после ответа); TTL 24 ч даёт окно доступа, затем логическая недоступность + best-effort sweep (§6).
- **Хранить картинки временного чата вечно (как обычные).** Отклонено: противоречит «временности» — картинка не должна переживать смысл временного чата бесконечно.
- **Периодический GC через cron/планировщик.** Отклонено: планировщика в проекте нет ([TD-010](../100-known-tech-debt.md)/[TD-013](../100-known-tech-debt.md)); opportunistic sweep на Postgres+Redis (§6) не требует новой инфраструктуры.

## Последствия

- **Положительные:** картинки хранятся вне чата, история лёгкая; фетч отдельным JWT-endpoint с изоляцией по владельцу; отдельный контролируемый биллинг; идемпотентность защищает от двойного списания/дубля строки при реплее; ход не падает на content-policy отказе; **картинки временного чата логически истекают по TTL безусловно (§6), не завися от планировщика**; **владелец может удалить своё изображение из галереи (`DELETE /v1/images/{imageId}` → `204`, §9), освобождая BYTEA; чужое/истёкшее/повторное → `404` без раскрытия существования**.
- **Цена:** таблица `generated_images` + миграция `0016` (**сразу с `expires_at`**); новый клиент `ImageGenerator`/`OpenAIImageGenerator`; tool `image.generate`; router `GET /v1/images/{id}` (с проверкой `expires_at`) **и `GET /v1/images` (список, курсорная пагинация, §8)** **и `DELETE /v1/images/{imageId}` (hard-delete, §9, амендмент 2026-07-14)** — **обе без новой миграции**; новый `blockReason=image_credits_empty`; opportunistic sweep (Redis-лок+троттлинг); config `IMAGE_*` + `TEMPORARY_IMAGE_TTL_SECONDS`/`IMAGE_SWEEP_*`.
- **Tech debt:** физический sweep истёкших картинок — best-effort без планировщика ([TD-036](../100-known-tech-debt.md)); `generated_images` в BYTEA, вынос в object storage — [TD-031](../100-known-tech-debt.md) (в ряд с [TD-009](../100-known-tech-debt.md)/[TD-027](../100-known-tech-debt.md)).
- **Безопасность/приватность:** доступ к байтам только владельцу (`user_id`), чужое **и истёкшее** → 404 (по `expires_at`, не по факту удаления); **удаление (§9) — только владельцу, owner+freshness в самом `DELETE`-запросе; чужое/истёкшее/несуществующее → 404 (число удалённых строк = 0), существование не раскрывается; повтор → 404, не 204**; `nosniff`/`no-store`; **промт хранится server-side (`generated_images.prompt`), НЕ возвращается в `images[]` ответа (§7, приватность/[TD-035](../100-known-tech-debt.md)) и не логируется** (как и байты, и ключ); списание идемпотентно по `tool_call_id` (нельзя обойти реплеем); картинка не сохраняется при неуплате; сбой генерации/невалидный ключ деградируют в tool_result-ошибку (не 502), инфра-сбой логируется (§3); временность изображений обеспечена логически (§6).
