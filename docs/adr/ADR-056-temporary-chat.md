# ADR-056 — Временный чат: без персистентности, история от клиента, только server-side инструменты

- **Статус:** Accepted
- **Дата:** 2026-07-09
- **Связано:** [ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md) (единственный писатель `chat_sessions`/`chat_steps`/`tool_calls` — `ChatRepository`; нормализация блоков, BUG-5), [ADR-011](ADR-011-server-side-tools.md) (server-side tools исполняются в tool-loop без round-trip), [ADR-025](ADR-025-parallel-tool-calls-and-max-tokens-truncation.md) (client-side барьер хода через continuation в БД), [ADR-023](ADR-023-sync-ids-in-chat-response.md) (`messageStepId`/`stepId` в ответе), [ADR-006](ADR-006-credit-billing-and-subscription-grant.md) (1 кредит = 1 сообщение, идемпотентность по `messageStepId`), [ADR-005](ADR-005-idempotency-ledger.md) (idempotency ledger), [ADR-059](ADR-059-openai-default-provider.md) (`store=False` в Responses)

## Контекст

В меню есть «временный чат» — беседа, которая **не сохраняется в истории и не используется для обучения AI**. Сейчас вся история безусловно пишется в `chat_sessions`/`chat_steps`/`tool_calls`; единственный писатель этих таблиц — `ChatRepository` (инвариант [ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md)). Нужен путь генерации, который ничего не персистит, но переиспользует существующий оркестратор, биллинг и policy.

## Решение

### 1. Ничего не пишется в БД; история приходит от клиента

При `temporary=True` в БД **не создаётся** ни `chat_sessions`, ни `chat_steps`, ни `tool_calls`. Так как сервер не хранит контекст, **историю присылает клиент** — поле `history: list[{role: "user"|"assistant", content: str}]`, провайдер-агностичный **текстовый транскрипт** реплик. Клиент не должен знать wire-формат провайдера (content-блоки Anthropic vs items OpenAI) — только роль и текст; оркестратор сам собирает из транскрипта сообщения для активного клиента.

Инвариант «только `ChatRepository` пишет chat_*-таблицы» ([ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md)) **сохраняется**: в temporary-ходе `repo` подменяется на in-memory реализацию с идентичными сигнатурами (`get_or_create_session` возвращает синтетический непersisted `ChatSession`; `list_steps` отдаёт историю, засеянную из `history`; все write-методы — no-op). Персистящий `ChatRepository` в этом пути не участвует.

### 2. Запрещены ТОЛЬКО client-side tools; server-side разрешены

Client-side инструменты (`files.*`/`calendar.*`/`reminders.*`) исполняются на устройстве и требуют **континьюации через БД**: backend возвращает `status=tool_call`, ждёт `/chat/tool-result`, восстанавливает состояние хода из `tool_calls`/`chat_steps` ([ADR-025](ADR-025-parallel-tool-calls-and-max-tokens-truncation.md)). Без персиста этот цикл невозможен. Поэтому во временном чате client-side tools **не предлагаются** модели (`include_client_side=False`).

Server-side инструменты (`time.now` [ADR-026](ADR-026-global-server-side-tools-and-time-now.md), `quiz.generate` [ADR-057](ADR-057-study-learn-quiz-contract.md), `image.generate` [ADR-058](ADR-058-image-generation.md), `site.*` при наличии проекта) исполняются backend **внутри одного запроса** в tool-loop ([ADR-011](ADR-011-server-side-tools.md)) — БД для континьюации не нужна, поэтому они **разрешены** во временном чате. Если client-side `tool_use` всё же прилетел от модели (аномалия, инструмент не предлагался) — трактуется как `UpstreamError`, симметрично существующему `site.*`-guard.

### 3. Взаимоисключения (валидатор запроса)

`temporary=True` несовместимо с любым, что подразумевает персист/резюме — все → `422`:

- `temporary` + `sessionId` → временный чат нельзя резюмировать (сессии в БД нет);
- `temporary` + `editMessageStepId` → редактирование требует усечения истории в БД ([ADR-040](ADR-040-edit-message-and-regenerate.md));
- `temporary` + `projectId` / `workspaceProjectId` → завязаны на персистентные сущности;
- `history` без `temporary` → history-транскрипт применим только к временному чату (обычный чат берёт историю из БД);
- размер `history` ≤ `settings.size_limit_context`.

### 4. Биллинг, trial, audit, sync-id при `session_id=None`

Сессии в БД нет ⇒ FK на `chat_sessions` нельзя проставить. Поэтому:

- `audit.record(session_id=None)` и `wallet.consume(session_id=None, ...)`. `wallet.consume` уже пропускает `_validate_session` при `session_id=None` — этот путь остаётся корректным.
- **Кредит за ход списывается как обычно** ([ADR-006](ADR-006-credit-billing-and-subscription-grant.md)): временный ≠ бесплатный. Ключ идемпотентности дебета — `messageStepId`, который **присутствует** в ответе (он минтится на каждый message-шаг [ADR-005](ADR-005-idempotency-ledger.md), не требует персиста сессии). `stepId` в ответе = `null` (persisted-шага нет, [ADR-023](ADR-023-sync-ids-in-chat-response.md)).
- **Пожизненный trial потребляется временным чатом** (реализация: `EphemeralChatRepository` **не** переопределяет `mark_trial_used`). Первый ход пользователя без подписки во временном чате тратит его lifetime-trial и делает реальный `UPDATE users SET trial_used = TRUE` ([ADR-002](ADR-002-access-policy-state-machine.md), BR-1 «один пожизненный trial»). **Почему так, а не «trial не тратится»:** иначе через `temporary=true` пользователь получил бы **неограниченные** бесплатные генерации, обойдя инвариант «один пожизненный trial». Запись идёт в `users` (булев флаг), **не** в chat-таблицы, поэтому инвариант единственного писателя chat_* ([ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md)) не нарушен; содержимое диалога не сохраняется — меняется только флаг.
- **Три следа временного хода в БД (важно для честности §5), ни один не содержит текста диалога:**
  1. **`users.trial_used`** — переключается в `TRUE`, если этим ходом был потреблён пожизненный trial (см. bullet выше).
  2. **`ledger_transactions`** — строка списания кредита за ход (`type='debit'`, `amount=1`, `meta` с usage). Привязана к `user_id` и `idempotency_key` (= `messageStepId` хода). **Ссылки на сессию не содержит вовсе** — у таблицы **нет** колонки `session_id` и нет FK на `chat_sessions` ([03-data-model.md §4 ledger_transactions](../03-data-model.md#4-ledger_transactions)), поэтому синтетический `sessionId` временного чата туда физически попасть не может.
  3. **`audit_logs`** — временный ход пишет **ТРИ** audit-события, все с `session_id = NULL`: **`policy_decision`** (`orchestrator.py`, до генерации), **`chat_step`** (assistant-шаг, `orchestrator.py` — пишется даже во временном ходе, хотя строки `chat_steps` нет), **`billing_debit`** (`wallet/service.py`). Во всех трёх `session_id` проходит через `_fk_session_id()`, который во временном ходе (`self._temporary=True`) возвращает `None` — **именно здесь обеспечивается инвариант «нет FK на несуществующую сессию»:** `audit_logs.session_id` — FK-несущая колонка (`ON DELETE SET NULL`), записать в неё синтетический непersisted uuid нельзя (INSERT упал бы), поэтому туда идёт `NULL`. **Метаданные хода фиксируются в audit, хотя переписка не хранится:** `chat_step.payload` несёт `role`, `model`, `usage` (токены) и **синтетический `sessionId`** (как строка в `payload.sessionId`, НЕ как FK); `billing_debit.payload` — `ledgerTxId`/`amount`/`newBalance`/`model`. **Текста диалога (сообщений/ответов) ни одно из трёх событий не содержит.**
  Все три вида следов (trial-флаг, ledger, три audit-записи) — **биллинговые/анти-абузные/policy-метаданные**, без текста диалога; они существуют именно потому, что временный ход реально стоит денег, реально расходует trial и проходит policy.
- **Идемпотентность дебета — только в пределах одного HTTP-запроса.** Так как `messageStepId` не персистится, повторный идентичный HTTP-запрос сгенерирует новый `messageStepId` и спишет заново. Это **by design** (у временного чата нет континьюации, нет retry-по-тому-же-ходу): в пределах одного запроса ledger-идемпотентность работает штатно, кросс-запросной идемпотентности нет — [TD-032](../100-known-tech-debt.md).

### 5. «Не хранится у нас» vs «не обучается» — честное разграничение

Две гарантии обеспечиваются **разными** механизмами, и это важно не путать:

- **«Не сохраняется в истории» (не хранится у нас)** — реализуется **нашим кодом**: ни одной строки в chat_*-таблицах (`chat_sessions`/`chat_steps`/`tool_calls`), `stepId=null`, `GET /v1/chats` беседу не покажет. Проверяемо прямым SQL-count после запроса. **Важная оговорка (не преувеличивать):** «не сохраняется» относится к **переписке**, а не к «нулю следов в БД» — временный ход оставляет **не-контентные** следы (см. §4): дебит в `ledger_transactions` (без ссылки на сессию), **три audit-события** (`policy_decision`/`chat_step`/`billing_debit`, все `session_id = NULL`; `chat_step` несёт метаданные хода — role/model/usage/синтетический sessionId, но **не текст**) и, при потреблении, флаг `users.trial_used`. Ни один след не содержит текста диалога. UI/маркетинг должны обещать «мы не храним переписку этого чата», **а не** «в базе не остаётся ничего»: факт списания, метаданные хода в audit и израсходованный trial фиксируются (иначе временный чат был бы бесплатным обходом лимитов).
- **Изображения временного чата — отдельное хранение с TTL ([ADR-058 §6](ADR-058-image-generation.md)).** Генерация изображений (`image.generate`, спринт 5) во временном чате **разрешена**, но байты пишутся в отдельную таблицу `generated_images` и **переживают** диалог. Чтобы это не противоречило «временности», такие изображения сохраняются с `expires_at = created_at + TEMPORARY_IMAGE_TTL_SECONDS` (дефолт 24 ч) и после истечения **логически недоступны** (`GET /v1/images/{id}` → `404`, как для чужого/несуществующего), а байты удаляются best-effort (opportunistic sweep). Честно: изображение временного чата живёт **до истечения TTL**, а не «исчезает сразу после хода». Изображения обычного чата — `expires_at = NULL` (не истекают). Детали — [ADR-058 §6](ADR-058-image-generation.md).
- **«Не используется для обучения AI»** — опирается на **условия API-договора провайдера**, а не на флаг в нашем запросе. Anthropic и OpenAI по условиям использования API **не обучают** модели на данных API-запросов — это уже так для всего трафика, независимо от `temporary`. **Асимметрия провайдеров (честно):** флаг `store=False` (провайдер не сохраняет данные запроса у себя) есть **только у OpenAI Responses** ([ADR-059](ADR-059-openai-default-provider.md)); у **Anthropic** аналогичного флага в нашем пути нет — там гарантия «не хранится» держится на **нашем не-персисте** плюс **data-policy провайдера** (Anthropic не обучает и не хранит на данных API по условиям), а не на явном флаге запроса. Мы **не** можем технически гарантировать «не обучается» кодом бэкенда ни на одном провайдере — это гарантия провайдера. Формулировка в UI/маркетинге не должна преувеличивать: честно — «мы не храним переписку этого чата; провайдер по условиям API не использует данные для обучения».

### 6. Context spoofing через клиентскую `history` — ПРИНЯТЫЙ риск

Клиент присылает `history` как произвольный транскрипт, включая реплики с `role="assistant"` — то есть может **«подделать» реплики ассистента** (вложить в историю ответы, которых модель не давала). Это **осознанно принято** как приемлемый риск, не дефект:

- **Только собственный диалог.** Пользователь манипулирует **своим** контекстом в **своём** запросе — cross-user влияния нет (история не шарится, изоляция по `sub` сохраняется).
- **Системный промпт серверный и выше по авторитету.** `history` попадает в user/assistant-сообщения, а не в `system`; серверный system-prompt ([ADR-037](ADR-037-chatrunrequest-context-allowlist-injection.md)/[ADR-042](ADR-042-hide-context-block-from-user-facing-history.md) не нарушены) сохраняет высший авторитет. Подделка assistant-реплик не даёт привилегий и не эскалирует доступ.
- **Симметрично обычному чату.** В обычном (persisted) чате история тоже в конечном счёте сформирована ходами пользователя; разница лишь в том, что там её хранит БД, а здесь присылает клиент. Ни там, ни там пользователь не может выйти за пределы собственного контекста.
- **Нет усиления ущерба.** Мутирующие client-side tools во временном чате отключены (§2); server-side мутации (`site.*`) требуют проекта/владельца; биллинг/policy подделкой истории не обходятся.

Вывод: context spoofing здесь — не уязвимость, а свойство «клиент владеет своим контекстом»; защищаться от того, что пользователь врёт сам себе в своём же диалоге, не требуется. TD не заводится.

## Альтернативы

- **Хранить временный чат с TTL/флагом `is_temporary` и фоновой очисткой.** Отклонено: «не хранится» превращается в «хранится недолго» — слабее гарантии; требует планировщика (которого нет, ср. [TD-010](../100-known-tech-debt.md)/[TD-013](../100-known-tech-debt.md)); всё равно виден в БД до очистки.
- **Разрешить client-side tools во временном чате.** Отклонено: невозможно без персиста (континьюация `/chat/tool-result` восстанавливает ход из `tool_calls`/`chat_steps`, [ADR-025](ADR-025-parallel-tool-calls-and-max-tokens-truncation.md)); эмуляция состояния хода в Redis/памяти между HTTP-запросами — новая инфраструктура ради узкого случая.
- **Присылать историю в wire-формате провайдера.** Отклонено: течёт провайдер-специфика на клиент (нарушает [ADR-033](ADR-033-llm-provider-abstraction.md)); текстовый транскрипт провайдер-агностичен и достаточен для контекста. Сырые `tool_use` без парного `tool_result` дали бы `400` от провайдера (BUG-5, [ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md)) — в транскрипт пускаются только текстовые user/assistant реплики.
- **Не списывать кредит за временный ход.** Отклонено: генерация стоит того же upstream, что обычная; «временный» — про приватность/историю, не про бесплатность. 1 кредит = 1 сообщение сохраняется ([ADR-006](ADR-006-credit-billing-and-subscription-grant.md)).

## Последствия

- **Положительные:** приватный режим без персиста; переиспользует оркестратор/policy/биллинг; инвариант единственного писателя chat-таблиц не нарушен ([ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md)); честная формулировка гарантий; server-side инструменты (в т.ч. квиз [ADR-057](ADR-057-study-learn-quiz-contract.md) и картинки [ADR-058](ADR-058-image-generation.md)) работают.
- **Цена:** новая in-memory `EphemeralRepository` с сигнатурами `ChatRepository` (mypy strict на `app.chat.*`); ветка подмены `repo` в `run(...)`; набор взаимоисключающих валидаций; client-side tools отключены.
- **Tech debt:** нет кросс-запросной идемпотентности дебета временного чата ([TD-032](../100-known-tech-debt.md)) — by design.
- **Безопасность/приватность:** данные хода не попадают в БД; `store=False` на стороне провайдера; гарантия «не обучается» явно атрибутирована провайдеру, не преувеличивается.
