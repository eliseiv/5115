# ADR-064 — Файлы-знания workspace: живая переинъекция на каждом ходе (а не «персист-и-реплей»)

- **Статус:** Accepted
- **Дата:** 2026-07-14
- **Тип:** bug-fix + ревизия решения (живой функциональный баг на проде, подтверждён эмпирически)
- **Ревизует (в части подачи ФАЙЛОВ-знаний):** [ADR-036 §3/§6](ADR-036-workspaces-implementation.md) (файлы «сохранены как content-блоки истории и реплеятся автоматически» — на самом деле никогда не персистились) и [ADR-038 §3.2/§3.3](ADR-038-move-chat-to-workspace.md) (вариант a — файлы turn-0-only, не подаются перенесённому чату). Тела ADR-036/038 не переписаны (immutability) — актуальное поведение подачи файлов здесь. Инъекция `instructions` (ADR-036 §3, ADR-038 §3.1) **не меняется**.
- **Связан с:** [ADR-020](ADR-020-inline-base64-attachments-mvp.md) (inline-base64, инвариант «сырой base64 не персистится»), [ADR-042](ADR-042-hide-context-block-from-user-facing-history.md) (утечка инъектированного контента в user-facing историю), [ADR-033](ADR-033-llm-provider-abstraction.md) (провайдер-агностичность), [ADR-040 §4а](ADR-040-edit-message-and-regenerate.md) (edit первого сообщения). Закрывает [Q-038-1](../99-open-questions.md), [Q-040-3](../99-open-questions.md).
- **Без миграции БД.** Меняется только логика подачи контекста в orchestrator (`src/app/chat/orchestrator.py`).

## Context

**Симптом (подтверждён на проде эмпирически).** Чат, привязанный к workspace (`workspace_project_id`), на ПЕРВОМ ходе корректно читает файлы-знания (модель отвечает фактом из файла). На ЛЮБОМ следующем ходе той же сессии агент теряет доступ к файлам и галлюцинирует: turn 0 — «столица — Осло» (верно, из файла), turn 1 на вопрос про другой факт того же файла — выдуманный ответ («белый медведь» вместо «выдра»). Файлы workspace доступны только на turn 0.

**Root cause (установлен по коду, `src/app/chat/orchestrator.py`).**
- turn 0 (`ctx.is_new`, ветка ~703–714): `context_for_session` собирает `(instructions, files)`; файлы кладутся в `workspace_attachments`.
- Персистимый user-step (~753–767): `user_payload_content = [*text_blocks, *placeholders]`, где `placeholders` — **только** плейсхолдеры вложений запроса (`prepared.placeholders`). Блоки файлов workspace сюда **не попадают**.
- `first_turn = _merge_attachments(prepared, workspace_attachments)` (~759) уходит только в ЖИВОЙ первый запрос (`first_turn_attachments`), в БД не пишется. `_merge_attachments` для workspace-файлов не создаёт плейсхолдеров.
- Продолжение (`not ctx.is_new`, ветка ~715–719): вызывается только `instructions_for_session` (инструкции без файлов); `context_for_session` **не** вызывается. Реплей истории из БД (`_build_messages`) файлов не содержит — их там нет.

Итог: файлы workspace физически отсутствуют в контексте всех ходов после turn 0. Это **прямо противоречит** нормативному тексту [ADR-036 §3/§6](ADR-036-workspaces-implementation.md) и [ADR-038 §3.2](ADR-038-move-chat-to-workspace.md): «Knowledge FILES … persisted as history content blocks on turn 0 and replayed automatically». Персист-и-реплей файлов **никогда не был реализован** — это docs↔code mismatch, материализовавшийся как функциональный баг.

**Дизайн-конфликт, который надо разрешить.** Нельзя «просто начать персистить блоки файлов в user-step», потому что:
1. **Инвариант хранения ([ADR-020](ADR-020-inline-base64-attachments-mvp.md), orchestrator ~727):** сырой base64 НИКОГДА не персистится в `chat_steps` (для вложений запроса есть лёгкие текстовые плейсхолдеры; сами блоки клиент подаёт один раз). Файлы-картинки workspace — это base64-data-URI (`input_image` / `image`); их персист в `chat_steps` нарушил бы инвариант. Текстовые файлы (`extracted_text` → `input_text`/`text`) инвариант НЕ нарушают.
2. **Утечка в user-facing историю ([ADR-042](ADR-042-hide-context-block-from-user-facing-history.md)):** если персистить текстовые блоки `[Файл проекта: {filename}]\n{extracted_text}` в user-step, они станут видны пользователю в `GET /v1/chats/{id}` и в превью `GET /v1/chats` — ровно тот класс дефекта, что ADR-042 уже чинил для conversation-settings блока (потребовался бы новый read-time strip по хрупкому формату).

Рассмотренные варианты подачи файлов после turn 0:
- **(A) Переинъекция файлов на каждом ходе** (звать `context_for_session` и на продолжении). Файлы всегда видны модели; ничего не персистится (оба инварианта целы). Стоимость: повторное чтение/декод файлов из `workspace_files` на каждый ход.
- **(B) Персист-и-реплей** (как заявляли ADR-036/038, но никогда не делали). Для текста реализуемо, но: (1) для картинок конфликт с base64-инвариантом (нужен плейсхолдер + гидрация из `workspace_files` на реплее — сложнее); (2) утечка текста файлов в user-facing историю → нужен новый ADR-042-подобный strip. Асимметрия «текст персистим / картинки нет» и новый strip — лишняя сложность и регрессионный риск.
- **(C) Гибрид** (текст персистим, картинки переинъектим). Наследует минусы (B): утечка текста в историю + новый strip + асимметрия трёх поведений (instructions live / текст-файлы snapshot / картинки live).

## Decision

Принят **вариант (A): файлы-знания workspace — ЖИВОЙ контекст, переинъектируемый на каждом ходе, НЕ персистируемый.** Симметрично с `instructions` (которые уже живут в `system` и переинъектируются на каждом обращении к LLM). Это единственный вариант, при котором оба инварианта (base64-хранение и отсутствие утечки в историю) сохраняются без нового strip-механизма, а картинки и текст ведут себя одинаково.

### 1. Файлы инъектируются в первый LLM-вызов КАЖДОГО generation-запроса сессии с workspace

Для сессии с непустым `sess.workspace_project_id` файлы-знания живого (на момент хода) workspace собираются через `WorkspacesService.context_for_session` и подаются в **первый** вызов `create_message` **каждого** запроса генерации — **независимо от `ctx.is_new`**:
- **turn 0** (новая сессия) — как сейчас (без изменений поведения);
- **resume / следующее сообщение** `/chat/run` (`not ctx.is_new`) — **новое**: раньше собирались только `instructions`, теперь и файлы;
- **первый continuation-вызов** `/chat/tool-result` — **новое**: раньше собирались только `instructions`, теперь и файлы.

Механика подачи — существующая (`first_turn_attachments` → клиент инжектит блоки в последний user-turn). Ветвление `if ctx.is_new / else` в `run()` для workspace **схлопывается** в единый путь `if sess.workspace_project_id is not None: context_for_session(...)`.

### 2. Файлы НЕ персистируются (оба инварианта целы)

- `user_payload_content` (персистимый user-step) по-прежнему = `[*text_blocks, *placeholders]` и **не** содержит блоков файлов workspace. Сырой base64 картинок в `chat_steps` не попадает ([ADR-020](ADR-020-inline-base64-attachments-mvp.md) — инвариант цел). Текст файлов не попадает в user-facing историю/превью → **новый strip (как ADR-042) НЕ требуется**.
- `_merge_attachments` продолжает отдавать `placeholders = list(chat_placeholders)` (только вложения запроса); файлы workspace плейсхолдеров не оставляют — без изменений.

### 3. Переиспользование turn-0-плюмбинга; расход после первого вызова

Внутри `_generate_loop` `turn0_attachments` обнуляется после первого `create_message` (существующее поведение, orchestrator ~1257). Поэтому файлы подаются в **первый** вызов запроса и **не** повторяются на внутренних server-side tool-раундах того же запроса — **паритет по стоимости** с прежним turn-0-поведением. Инвариант формулируется как «файлы workspace присутствуют в первом LLM-вызове каждого generation-запроса сессии с workspace».

### 4. Живое состояние workspace; graceful на удалении

Файлы читаются из **текущего** состояния `workspace_files` на каждый ход (не снимок). Удалённый/опустошённый workspace или отсутствие файлов → инъекции нет (обычный чат, graceful — как `context_for_session` возвращает `None`/пустые attachments сейчас). Провайдер-агностично ([ADR-033](ADR-033-llm-provider-abstraction.md)): текст `extracted_text`, картинки — vision-блок через клиент.

### 5. Разведение с inline-вложениями запроса ([ADR-020](ADR-020-inline-base64-attachments-mvp.md)) — ЯВНО

- **Вложения запроса** (`attachments` в `/chat/run`) — **turn-0-only**, одноразовый пользовательский ввод: base64 в живом первом запросе хода, лёгкий плейсхолдер персистится, на последующих ходах модель видит только плейсхолдер. **Без изменений.**
- **Файлы-знания workspace** — **живой проектный контекст**, переинъектируется на КАЖДОМ ходе, НЕ персистируется вовсе.

Прежняя формулировка «файлы workspace следуют модели inline-attachments (тяжёлый контент один раз)» ([ADR-036 §6](ADR-036-workspaces-implementation.md)) **снята**: workspace-файлы — долгоживущий контекст проекта, а не одноразовый ввод, поэтому симметрия проводится не с «один раз», а с `instructions` («живой контекст на каждом ходе»).

### 6. Перенесённые и отредактированные чаты — файлы теперь применяются (закрытие Q-038-1 и Q-040-3)

Поскольку подача файлов развязана от `ctx.is_new`, единый путь `run()` срабатывает для **любой** сессии с workspace, как бы она его ни получила:
- **Перенесённый чат** (`PATCH /v1/chats/{id}` с `workspaceProjectId`, [ADR-038](ADR-038-move-chat-to-workspace.md)): со следующего сообщения получает и `instructions`, и **файлы**. **[Q-038-1](../99-open-questions.md) закрывается в пользу варианта b** (реинъекция файлов), т.к. живая переинъекция делает это тривиальным и соответствует продуктовому смыслу переноса.
- **Отредактированное первое сообщение** ([ADR-040 §4а](ADR-040-edit-message-and-regenerate.md), `is_new=False` после усечения): следующий ход снова инъектирует файлы. **[Q-040-3](../99-open-questions.md) закрывается** (та же ось).

Это **суперсидит** решение [ADR-038 §3.2 вариант a](ADR-038-move-chat-to-workspace.md) и строку «files, resume/continuation → нет» таблицы §3.3.

### 7. Обоснование против аргументов стоимости ADR-038 §3.2

Аргумент [ADR-038 §3.2](ADR-038-move-chat-to-workspace.md) о стоимости токенов «файлы на каждом ходе» опирался на неверную предпосылку: он неявно предполагал, что «персист-и-реплей» дешевле, чем «переинъекция». На деле реплей истории отправляет весь накопленный контекст (включая файлы, если бы они там были) в **каждый** запрос — то есть «файлы в каждом запросе» — неизбежная стоимость самой фичи «знания доступны на каждом ходе», и обе схемы её платят одинаково. Единственная **дополнительная** стоимость варианта (A) — backend-side повторное чтение/декод файлов из `workspace_files` (см. TD-040), ограниченное лимитами `WORKSPACE_FILE_MAX_COUNT`/`WORKSPACE_FILES_TOTAL_BYTES`/`WORKSPACE_CONTEXT_MAX_CHARS` ([ADR-036 §4/§6](ADR-036-workspaces-implementation.md)).

**Prompt-cache:** `cache_control: ephemeral` в этом коде стоит только на `system` (orchestrator ~102–111). Файлы живут в user-content и не кэшируются ни при персист-и-реплее, ни при переинъекции → регрессии кэша нет. `system` (base + `instructions` + dialog-mode суффикс) — без изменений, стабилен между ходами при неизменных `instructions`.

### 8. Биллинг, миграция, безопасность

- **Биллинг неизменен** ([ADR-006](ADR-006-credit-billing-and-subscription-grant.md)): 1 кредит = 1 сообщение; инъекция файлов на amount не влияет.
- **Миграции нет** — только логика orchestrator.
- **Безопасность:** файлы читаются строго для владельца (`WHERE user_id = :sub` в repository); модель угроз инъекции пользовательского контента — [05-security.md §Workspaces](../05-security.md), неизменна. Отсутствие персиста файлов не снижает безопасность (изоляция — на чтении).

## Указания backend (ТЗ)

Файл: `src/app/chat/orchestrator.py` (+ переиспользование `src/app/workspaces/service.py`).

1. **`run()` — workspace-ветка (~703–719).** Схлопнуть `if ctx.is_new / else` в единый путь: при `sess.workspace_project_id is not None` вызывать `context_for_session(sess.workspace_project_id, user_id, provider=_active_provider())` **независимо** от `ctx.is_new`. Из результата: `system_prompt = _system_prompt_with_workspace(sess.assistant_mode, ws_context.instructions)` и `workspace_attachments = ws_context.attachments`. Дальнейший `first_turn = _merge_attachments(prepared, workspace_attachments)` и `first_turn_attachments=first_turn` — без изменений (файлы уходят в живой первый вызов). `None` из `context_for_session` (удалён/чужой workspace) → base system-prompt, без файлов (graceful).
2. **НЕ персистить файлы.** `user_payload_content` оставить `[*text_blocks, *placeholders]` — блоки файлов workspace в персист НЕ добавлять (инвариант base64 + отсутствие утечки в историю). `_merge_attachments` не менять (placeholders только из вложений запроса).
3. **`tool_result()` — continuation (~905–910).** Заменить `instructions_for_session` на `context_for_session` (собрать `instructions` + файлы); `system_prompt` — из `ws_context.instructions`; передать в `_generate_loop` **`first_turn_attachments=_merge_attachments(None, ws_context.attachments)`** (сейчас этот вызов `_generate_loop` не передаёт `first_turn_attachments`). Так файлы попадут в первый вызов continuation. Graceful при `None`.
4. **`instructions_for_session`** становится избыточным (оба call-site теперь идут через `context_for_session`). Решение исполнителя: удалить метод (+ его тесты/ссылки) ИЛИ оставить неиспользуемым; обосновать выбор. Не оставлять мёртвый вызов.
5. **Инвариант расхода:** `turn0_attachments` обнуляется после первого `create_message` (~1257) — не менять; файлы не повторяются на server-раундах (паритет стоимости).
6. **Обработка ошибок `context_for_session`:** метод по сигнатуре возвращает `WorkspaceContext | None` и не бросает доменных исключений (при отсутствии/чужом workspace → `None`; при пустых файлах → `attachments=None`). Обрабатывать оба случая как «без файлов, base system-prompt». Иных классов исключений у этой ветки нет; исключения инфраструктуры БД пробрасываются как обычно (на усмотрение исполнителя — не глушить).
7. **Тест (обязательно, многоходовой).** turn 0: workspace c текстовым файлом, содержащим факт A и факт B; вопрос про факт A → ответ A. turn 1 (**resume той же сессии**, `is_new=False`): вопрос про факт **B**, которого НЕТ в предыдущей переписке (только в файле) → ассистент отвечает фактом B. Проверять на фейковом/stub LLM-клиенте, что блок файла присутствует в `attachments`/`messages`, переданных в `create_message` на turn 1. Добавить вариант перенесённого чата: обычный чат (turn 0 без workspace) → `PATCH` в workspace с файлом → turn 1 отвечает фактом из файла. Опционально: continuation-вариант (`/chat/tool-result` первый вызов несёт блок файла).

## Consequences

- **Фикс бага:** файлы-знания доступны модели на КАЖДОМ ходе (текст и картинки), а не только на turn 0. docs↔code mismatch устранён.
- **Инварианты целы:** сырой base64 не персистится ([ADR-020](ADR-020-inline-base64-attachments-mvp.md)); текст файлов не утекает в user-facing историю/превью → новый strip ([ADR-042](ADR-042-hide-context-block-from-user-facing-history.md)) не нужен.
- **Единая модель:** `instructions` И файлы — живой per-turn workspace-контекст (не персист); одна ментальная модель вместо трёх поведений. `context_for_session` — единственная точка сборки контекста; `instructions_for_session` устраняется.
- **Q-038-1 и Q-040-3 закрыты** (файлы применяются к перенесённым и отредактированным чатам).
- **Новый tech-debt — [TD-040](../100-known-tech-debt.md):** повторный декод base64 картинок из `workspace_files` на каждом ходе — избыточная backend-работа (ограничена лимитами файлов; оптимизация — кэш декодированных блоков в пределах запроса / пропуск при неизменном наборе).
- **Известное ограничение (не долг):** на внутренних server-side tool-раундах одного запроса файлы присутствуют только в первом вызове (расход `turn0_attachments`) — как и прежде для turn 0. Если понадобится — будущее усиление, аддитивно.
- **Контракты обновлены этим проходом (architect):** [chat-orchestrator/02-api-contracts.md](../modules/chat-orchestrator/02-api-contracts.md), [workspaces/03-architecture.md](../modules/workspaces/03-architecture.md); revision-пометки к ADR-036/038 в [INDEX](INDEX.md#ревизии). Backend приводит код в соответствие.

## Alternatives

- **(B) Персист-и-реплей текста + гидрация картинок.** Отвергнуто: утечка текста файлов в user-facing историю → новый ADR-042-подобный strip (хрупкая привязка к формату `[Файл проекта:]`); для картинок — плейсхолдер + гидрация из `workspace_files` (сложнее); асимметрия «текст snapshot / картинки live».
- **(C) Гибрид (текст персист, картинки переинъект).** Отвергнуто: наследует утечку текста в историю и новый strip; три разных поведения контекста.
- **Оставить как есть (файлы turn-0-only).** Отвергнуто: это и есть баг — для базы знаний недопустимо, чтобы знания исчезали после первого сообщения.
