# ADR-062 — Study & Learn: пул вопросов квиза + чистый assistantMessage

- **Статус:** Accepted
- **Дата:** 2026-07-13
- **Связано:** [ADR-057](ADR-057-study-learn-quiz-contract.md) (исходный контракт квиза — **пересматривается**: §3 «одна карточка» и §5 «`QuizSchema` — плоская карточка» superseded этим ADR; §2 function-tool/strict, §4 gating `study_learn`/global server-side, §6 client-side `correctIndex` — **в силе, не тронуты**), [ADR-055](ADR-055-dialog-mode.md) (режим `study_learn`), [ADR-059](ADR-059-openai-default-provider.md) (плоский `FunctionToolParam`, `strict`, очистка strict-схемы), [ADR-026](ADR-026-global-server-side-tools-and-time-now.md) (global server-side tool, degrade-прецедент `invalid_timezone`), [ADR-011](ADR-011-server-side-tools.md) (`MAX_SERVER_TOOL_ROUNDS` guard), [ADR-024](ADR-024-history-payload-domain-normalization.md) (`assistantMessage` при `tool_call` — **частично переопределяется** для квиз-хода, §3 ниже), [ADR-056](ADR-056-temporary-chat.md) (совместимость с временным чатом)

## Контекст

Реализация квиза Study & Learn ([ADR-057](ADR-057-study-learn-quiz-contract.md)) на практике дала UX-проблемы (подтверждено кодом и владельцем):

1. **Один вопрос вместо пула.** Схема `quiz.generate` — ОДНА карточка (`{question, options, correctIndex, explanation}`), аккумулятор отдаёт последнюю (`quiz_acc[-1]`). Пользователь получает ровно один вопрос, тогда как по макету нужен **пул** из нескольких вопросов сразу.
2. **Дублирование вопроса и спойлер ответа в тексте.** Описание инструмента предписывает модели «Keep writing your normal teaching text as well: the quiz complements the explanation» — модель **повторяет** вопрос в `assistantMessage` и **сразу раскрывает** правильный ответ там же. Пользователь видит ответ до того, как ответил, — спойлер.
3. **Целевой флоу по макету:** пользователю **сразу приходит пул** вопросов; он отвечает на все локально; результат показывается на клиенте и **не** отправляется обратно агенту (нет continuation-эндпоинта проверки ответов).

Нужно: (а) пул вопросов за один ход; (б) `assistantMessage` без дубля вопросов и без спойлера; (в) зафиксировать, что результат считается на клиенте и никуда не шлётся.

Это **breaking change** wire-контракта поля `quiz`. Реальных пользователей на релизе нет — миграция iOS на `quiz.questions[]` допустима без версионирования.

## Решение

### 1. Пул вопросов в ОДНОМ вызове `quiz.generate` (список, не N вызовов)

Аргументы инструмента `quiz.generate` меняются с одной карточки на **обёртку со списком вопросов**:

```
{ questions: [ { question, options, correctIndex, explanation }, ... ] }
```

- `questions` — список из **3..10** вопросов (нижняя граница 3 = осмысленный «пул», верхняя 10 = ограничение по токенам/латентности). Каждый элемент — прежняя карточка: `question` (≤1000), `options` (**2..10** вариантов, каждый ≤400), `correctIndex` (0-based, `< len(options)`), `explanation` (≤2000). Пороги длин и число вариантов **не меняются** относительно ADR-057.
- Все поля `required`, `additionalProperties:false` (strict, [ADR-059 §1](ADR-059-openai-default-provider.md)) — как на обёртке, так и на **каждом** вложенном вопросе (иначе провайдер даст `400`, см. §5).
- Числовые/списочные ограничения (`3..10` вопросов, `2..10` вариантов, длины) **невыразимы в strict-схеме** и остаются **серверной Pydantic-валидацией** + продублированы словами в описании инструмента (см. §5) — тот же принцип, что [ADR-057 §2/§3](ADR-057-study-learn-quiz-contract.md).

**Почему список в одном вызове, а не «модель зовёт tool N раз и мы аккумулируем».** Выбран **список в одном вызове**:
- Детерминированно «сразу пул»: клиент получает весь пул одним объектом за один ход — ровно как в макете.
- Проще клиент (один `quiz` с `questions[]`) и проще аккумулятор (маппинг args→schema, без склейки).
- Каждый отдельный вызов `quiz.generate` расходует server-side round tool-loop'а ([ADR-011 §2](ADR-011-server-side-tools.md)); N вызовов приближают к `MAX_SERVER_TOOL_ROUNDS` и удваивают латентность/риск, что модель вставит спойлер-текст между вызовами.
- Отвергнуто «N вызовов»: недетерминированное число, больше раундов, сложнее гарантировать «сразу пул».

### 2. Отдача клиенту: `ChatResponse.quiz` несёт пул

`QuizSchema` меняется с плоской карточки на **обёртку**: `{ questions: [QuizQuestionSchema] }`, где `QuizQuestionSchema = { question, options, correctIndex, explanation }`. Валидатор `correctIndex < len(options)` применяется **к каждому** вопросу. `ChatResponse.quiz: QuizSchema | null` — форма поля меняется (`quiz.questions[]`), присутствие/семантика (только при `dialogMode=study_learn` и вызванном `quiz.generate`) **не меняются**. Хранение — `chat_steps.payload` как tool-результат, без таблицы/миграции (как [ADR-057 §5](ADR-057-study-learn-quiz-contract.md)).

### 3. `assistantMessage` без дублирования и без спойлера — детерминированная гарантия

Проблема спойлера/дубля — текстовое поведение модели («текст зеркалит квиз»). Надёжно вырезать спойлер из свободного текста семантически невозможно. Поэтому — **два уровня, hard-уровень детерминированный**:

- **Промпт (defense-in-depth, мягкий).** Переписываются описание инструмента `quiz.generate` (SSOT `TOOL_DESCRIPTIONS`) и системный суффикс режима `study_learn`: все вопросы — **только** через `quiz.generate`; в тексте **не** повторять формулировки вопросов и **не** раскрывать/намекать правильные варианты и пояснения; сопроводительный текст держать коротким (краткая рамка/интро) или не писать вовсе.
- **Серверная гарантия (hard, нормативная).** **Когда в ответе присутствует `quiz` (не `null`), backend принудительно выставляет `assistantMessage = null`**, отбрасывая свободный текст модели этого хода. Это 100% устраняет обе жалобы (дубль вопроса и спойлер ответа): текст модели до пользователя не доходит; весь контент несёт структурированный `quiz.questions[]`, а рамку/интро рисует клиент. Гарантия применяется в единственной точке маппинга `ChatRunOut → ChatResponse` (общий helper роутера) и покрывает все статусы, где `quiz` присутствует (в т.ч. `assistant_message` и `tool_call`).
- **Взаимодействие с [ADR-024](ADR-024-history-payload-domain-normalization.md).** Правило «на `tool_call` пробрасывать сопутствующий текст в `assistantMessage`» **переопределяется только** для квиз-хода: при наличии `quiz` текст не пробрасывается (`assistantMessage=null`). История `GET /v1/chats/{id}` не меняется — сырой assistant-шаг (с его текстом, если он был) хранится/реплеится как есть; переопределяется лишь run-проекция ответа. Прочие (не-квиз) ходы `assistantMessage` не затрагиваются.

Отвергнуто: (а) фиксированная серверная строка-рамка вместо `null` — риск несовпадения языка с диалогом (рамку лучше рисует клиент своей локализованной копией); (б) частичное вырезание спойлера из текста — семантически ненадёжно, противоречит приоритету надёжности.

### 4. Результат считается на клиенте и НЕ уходит агенту (нормативно)

`correctIndex` и `explanation` каждого вопроса остаются в структурированном `quiz.questions[]` — клиент проверяет ответы **локально после** того, как пользователь ответил на все вопросы. **Эндпоинта отправки ответов НЕТ**; клиент ничего не шлёт назад, никакой continuation по результатам квиза не вводится. Это нормативное свойство контракта (не «MVP-упрощение, которое позже дополнится continuation»): подтверждает и ужесточает [ADR-057 §6](ADR-057-study-learn-quiz-contract.md) для пула. Серверная проверка ответов с «скрытым» `correctIndex` — отдельная будущая фича (серверное хранение состояния квиза), не долг.

### 5. Сохранение strict-wire-схемы (без `minItems`/`maxItems`/`minLength`/`maxLength`)

`quiz.generate` остаётся в `_OPENAI_STRICT_TOOLS`; его схема очищается `_clean_for_openai_strict` (allowlist [ADR-059](ADR-059-openai-default-provider.md)). Вложенный список объектов Pydantic сериализует через `$defs` (подмодель вопроса) + `$ref` в `questions.items` + `type: array` — все эти ключи в allowlist и **переживают** очистку; ограничивающие ключи (`minItems`/`maxItems`/`minLength`/`maxLength`) **срезаются** и держатся Pydantic-слоем. **Инвариант strict:** подмодель одного вопроса **обязана** быть strict-моделью (`extra='forbid'`), чтобы её запись в `$defs` несла `additionalProperties:false` и все свойства в `required` — иначе провайдер даст `400 Invalid schema` на каждом ходе. Не добавлять в strict-схему ограничивающие ключи — ужесточать только Pydantic-валидатором + текстом описания.

### 6. Аккумулятор: last-wins по пулу

`quiz_acc` аккумулирует **полные пулы** (dict `{questions:[...]}`) по раундам; в ответ идёт `quiz_acc[-1]` — **последний пул хода побеждает**. Механика не меняется относительно [ADR-057 §5](ADR-057-study-learn-quiz-contract.md) — меняется лишь форма элемента (пул вместо карточки). Если модель (вопреки инструкции «один вызов») вызвала `quiz.generate` несколько раз за ход — **последний полный пул побеждает** (не объединяем): детерминированно, без склейки возможно-дублирующихся вопросов, согласовано с текущей семантикой `quiz_acc[-1]`.

### 7. Обработка ошибок пула — all-or-nothing degrade

Режим отказа [ADR-057 §3](ADR-057-study-learn-quiz-contract.md) (graceful degrade: невалидные args → `tool_result`-ошибка `invalid_quiz`, модель исправляет в том же ходе, **не** `422`; при упорстве до `MAX_SERVER_TOOL_ROUNDS` — единый guard [ADR-011 §2](ADR-011-server-side-tools.md) → `UpstreamError → 502`) **сохраняется** и распространяется на пул как **all-or-nothing**:

- **Любое** нарушение делает **весь пул** невалидным → один `invalid_quiz` tool_result: вопросов вне диапазона `3..10` (в т.ч. **пустой список** и слишком длинный); у **любого** вопроса — число вариантов вне `2..10`, over-length `question`/`option`/`explanation`, `correctIndex` булев/отрицательный/вне `[0, len(options))`. Модель перегенерирует **весь** пул в том же ходе.
- **Частичное принятие отвергнуто** (выкинуть плохой вопрос, оставить хорошие): молча ужимает пул ниже запрошенного числа, усложняет аккумулятор, модель не знает какой вопрос выпал. All-or-nothing проще и держит пул когерентным.
- Технически: обёртка + список — **одна** Pydantic-модель; `model_validate` бросает **один** `ValidationError` на любое нарушение (вложенное включительно) → существующий единственный `except ValidationError` в `_quiz_generate` покрывает **все** классы (пустой список, вне-диапазона число вопросов/вариантов, длины, булев/отрицательный/вне-диапазона `correctIndex`) — иных классов исключений этот путь не бросает. Content-free reasons ([TD-035](../100-known-tech-debt.md)) строятся из `loc`+`type` (вложенный `loc` вида `questions.2.correctIndex` — только индексы/имена полей/код ошибки, без значений); текст degrade-сообщения обновляется под лимиты пула (`3-10 questions`, …).

### 8. Breaking change и миграция клиента

Форма `ChatResponse.quiz` меняется с плоской карточки на `{questions:[...]}` — **breaking**. Реальных пользователей на релизе нет; iOS-клиент **переходит на `quiz.questions[]`** (итерирует пул, локальная проверка каждого вопроса по `correctIndex`, показ `explanation` после ответа). Версионирование поля не вводится.

## Альтернативы

- **N вызовов `quiz.generate` + аккумуляция в пул.** Отвергнуто (§1): недетерминированное число вопросов, больше server-side раундов и латентности, риск спойлер-текста между вызовами.
- **Фиксированная серверная строка-рамка в `assistantMessage`.** Отвергнуто (§3): риск неверного языка; рамку лучше рисует клиент.
- **Частичное вырезание спойлера/дубля из текста модели.** Отвергнуто (§3): семантически ненадёжно; приоритет — надёжность.
- **Частичное принятие пула (drop bad question).** Отвергнуто (§7): молча ужимает пул, усложняет degrade, теряет обратную связь модели.
- **Расширить ADR-057 новой секцией вместо нового ADR.** Отвергнуто: пул переопределяет нормативные факты §3 (форма схемы) и §5 (`QuizSchema`) — по конвенции репозитория (immutability тела принятого ADR; superseding отдельным ADR, как [ADR-054](ADR-054-cloudpayments-webhook-payment-verification.md)↔[ADR-050](ADR-050-cloudpayments-webhook.md)) чистее отдельный ADR с явной пометкой пересмотра в шапке ADR-057 и записью в INDEX §Ревизии.

## Последствия

- **Положительные:** пул вопросов сразу одним ходом (макет); детерминированно нет спойлера/дубля (`assistantMessage=null` при квизе); клиент считает результат локально, без нового эндпоинта; переиспользует механику global server-side tool и degrade-путь ADR-057; strict-схема сохранена (вложенный `$defs`/`$ref`); без БД/миграции/доп-биллинга (1 кредит, [ADR-006](ADR-006-credit-billing-and-subscription-grant.md)).
- **Цена:** breaking-изменение формы `quiz` (iOS мигрирует на `quiz.questions[]`); потеря свободного текста модели на квиз-ходе (осознанно — контент несёт `quiz.questions[]`, рамку рисует клиент); переопределение ADR-024 для квиз-хода.
- **Безопасность:** без изменений относительно ADR-057 — `correctIndex` намеренно клиентский (обучение, не экзамен); квиз-текст не логируется ([TD-035](../100-known-tech-debt.md)); биллинг неизменен; инструмент не мутирует и не тарифицируется.
